# 从 Pi Agent Core 看下一代 Agent Runtime 的设计逻辑

最近在看 `Pi Agent Core` 的实现时，我有一个很强烈的感受：下一代 Agent 系统真正拉开差距的地方，已经不是“能不能接大模型”和“能不能调工具”，而是 **运行时本身到底怎么设计**。

很多项目把 Agent 理解成一层 Prompt 包装，再加一个工具调用循环；但一旦进入真实工程场景，问题就会马上变成：

- 消息上下文怎么裁剪，才不会越跑越重？
- 工具调用前后，哪些地方允许注入业务逻辑？
- 用户中途插一句话，Agent 应该立刻响应，还是等当前回合结束？
- 一次任务如果被打断，能不能从当前状态继续跑，而不是整局重来？

这些问题，本质上都不是 Prompt 工程问题，而是 Runtime 设计问题。`Pi Agent Core` 有意思的地方，就在于它开始认真回答这些问题。

## 1. 它不是“再包一层 LLM”，而是在做 Agent Runtime

从官方仓库的描述来看，`@earendil-works/pi-agent-core` 的定位非常明确：它是一个带有 **tool calling** 和 **state management** 的 Agent runtime，而不是单纯的模型 SDK。

这件事很重要，因为它意味着系统边界发生了变化：

- 模型调用只是其中一个环节
- 工具执行只是其中一个环节
- 真正的核心，是如何管理“消息、状态、队列、回合、终止条件”

换句话说，Pi 并不是把 Agent 当成“会调用工具的聊天机器人”，而是把它当成一个 **可持续运行、可中断、可恢复、可插桩的状态机**。

## 2. 第一层设计：内部消息和 LLM 消息是两套模型

`Pi Agent Core` 里有一个我很喜欢的设计：**内部一直使用 `AgentMessage[]`，直到真正调用模型时，才转换成 LLM 需要的 `Message[]`**。

这带来两个直接好处：

### 好处一：内部状态不会被模型接口绑死

系统内部可以保留更多语义，例如：

- 用户消息
- 助手消息
- 工具调用消息
- 工具结果消息
- 流式中间状态

这些信息对 Agent Runtime 很关键，但并不是所有模型供应商都原生支持。如果太早就把内部消息压扁成 provider message，很多运行时能力就很难做。

### 好处二：上下文治理终于有了正式插口

Pi 在调用模型前提供了两个分离的阶段：

1. `transformContext`
2. `convertToLlm`

这两个名字看起来很像，但其实解决的是两类完全不同的问题。

- `transformContext` 负责处理 **Agent 内部上下文**
- `convertToLlm` 负责处理 **对接具体模型接口**

我觉得这是一个非常成熟的拆分方式。

因为很多团队一开始会把“摘要压缩”“消息裁剪”“provider 适配”“role 转换”全堆在一个函数里，最后代码会越来越难维护。Pi 把它们拆开之后，职责就清楚了：

- 上下文怎么变短，是 Runtime 关注的事
- 最终怎么适配 OpenAI / Anthropic / Google，是模型边界的事

这相当于把 **context engineering** 和 **model adapter** 解耦了。

## 3. 第二层设计：Agent 循环不是单 while，而是“回合 + 队列”的双层结构

Pi 的 loop 设计，我觉得是它最值得借鉴的一部分。

从实现看，它不是简单地做：

```ts
while (true) {
  callLLM();
  callTools();
}
```

而是更接近下面这种思路：

```ts
while (true) {
  while (还有工具调用 或 还有待处理消息) {
    处理 steering 消息
    流式生成 assistant 回复
    执行工具调用
    判断是否提前终止
  }

  检查 follow-up 消息
  如果没有新的 follow-up，就结束
}
```

这个差异看起来不大，但实际意义非常大。

### `steering` 解决的是“当前流程里插一句话”

Pi 允许在当前 Agent 运行过程中，向队列里插入 steering message。它的语义更像：

> 当前这轮结束后，下一轮立刻把这条消息喂进去。

这很适合：

- UI 里用户补充说明
- 外部系统注入约束
- 任务执行中追加新的上下文

### `followUp` 解决的是“本来该结束了，但我想让你继续”

与之对应，`followUp` 只会在 Agent 原本准备停止时触发。它不是“打断”，更像“续一段”。

这很适合：

- 审批流后置动作
- 任务完成后的补充检查
- 异步结果回来后的再次推进

我认为这两个队列的区分非常工程化。因为它不是笼统地说“支持中途插消息”，而是明确区分：

- **影响当前流程节奏的消息**
- **影响结束后下一步动作的消息**

这背后其实是在设计 Agent 的控制平面。

## 4. 第三层设计：Agent 要能恢复，不只是能开始

很多 Agent Demo 只重视“如何启动一次任务”，但真实系统里更常见的问题是：

- 网络断了
- 用户刷新页面了
- 工具执行到一半失败了
- 模型接口超时了
- 当前回合结束后，想从已有上下文继续跑

Pi 提供了 `agentLoopContinue` / `runAgentLoopContinue` 这样的续跑能力，这说明它从一开始就在考虑 **continuation**。

这类设计对生产系统特别关键，因为它意味着：

- Agent 运行不一定要和某次页面会话强绑定
- 任务可以切片执行
- 状态可以序列化后再恢复
- 重试不必重复消耗之前所有 token

如果一个 Agent Runtime 只能从头跑，那它更像“智能脚本”；如果它能恢复和继续，它才开始像“长生命周期任务系统”。

## 5. 第四层设计：工具执行前后都要留 Hook

Pi 在工具执行前后分别预留了 `beforeToolCall` 和 `afterToolCall`。

这个设计看起来像小细节，但一旦进入业务系统，会非常有价值。

### `beforeToolCall` 适合做什么

- 参数校验
- 风险拦截
- 权限判断
- 补充审计信息
- 把自然语言参数改造成内部结构化字段

### `afterToolCall` 适合做什么

- 标准化工具返回
- 把脏数据清洗成可继续推理的结果
- 把结果写入观测系统
- 决定是否追加解释消息

这意味着 Pi 不强迫你把所有业务逻辑都塞进 Tool 里，而是允许你围绕 Tool 做一个更完整的执行管线。

在我看来，这类 Hook 的意义并不只是“方便扩展”，而是让 Runtime 真正有机会接入企业系统里的：

- 安全
- 合规
- 审计
- 可观测性

## 6. 第五层设计：状态不仅要给代码读，还要给 UI 读

Pi 的 `Agent` 不只是跑循环，它还维护一份明确的状态：

- `messages`
- `tools`
- `isStreaming`
- `streamingMessage`
- `pendingToolCalls`
- `errorMessage`

这说明它不是只面向 CLI 脚本，而是默认自己会被：

- Web UI 消费
- TUI 消费
- 调试器消费
- 上层 orchestration 系统消费

这点和很多“能跑起来”的 Agent 封装有一个本质区别：

> 很多实现只关心结果，Pi 更关心运行过程是否可被观察。

而在真实产品里，**过程可见性往往比最终答案更重要**。  
因为用户真正想知道的不只是“Agent 做完了吗”，而是：

- 它现在在想什么？
- 它卡在哪个工具上？
- 是不是还在流式输出？
- 失败是模型错了，还是工具挂了？

没有这些状态，Agent 产品几乎不可能做得稳定。

## 7. 这类架构给我们的启发是什么

如果把 Pi Agent Core 的设计抽象一下，我觉得它至少给了我们 4 个很明确的信号。

### 信号一：Agent Runtime 正在从 Prompt 层下沉到系统层

未来真正有壁垒的，不会只是 Prompt 模板，而是：

- 上下文治理
- 状态机设计
- 队列调度
- 工具执行管线
- 可恢复运行

### 信号二：上下文处理必须前置成一等能力

`transformContext` 这种插口说明了一件事：上下文压缩、裁剪、摘要、选择性保留，不能再作为“优化项”后补，而应该一开始就进入架构层。

### 信号三：Agent 的控制平面会越来越重要

`steering`、`followUp`、`prepareNextTurn`、`shouldStopAfterTurn` 这类能力，本质上都不是“让模型更聪明”，而是 **让系统更可控**。

在复杂场景里，可控性往往比单次回答质量更值钱。

### 信号四：下一代 Agent 更像任务运行时，不像聊天窗口

当一个系统开始认真设计：

- 如何开始
- 如何继续
- 如何暂停
- 如何恢复
- 如何观察
- 如何插入外部控制

它就已经不是普通 chatbot 了，而是在向 **agent runtime / task runtime** 演进。

## 8. 我会怎么借鉴这套思路

如果让我把这套思路迁移到自己的项目里，我会优先借鉴下面几件事：

1. 在内部保留 richer message model，不要太早降级成 provider message。
2. 把“上下文变换”和“模型适配”拆成两个阶段。
3. 把“用户插话”和“任务结束后续跑”当成两种不同队列。
4. 工具执行前后都预留 hook，而不是把所有逻辑压进 tool handler。
5. 把 streaming、pendingToolCalls、errorMessage 这类字段明确建模，方便 UI 和调试。

这些能力单独看都不炫，但组合起来，才会让 Agent 系统从“能演示”变成“能上线”。

## 结语

我越来越觉得，下一代 Agent 系统的竞争点，不会只是在模型能力上卷，而是在 Runtime 这一层卷。

谁能把以下几件事做好，谁的 Agent 才更可能稳定进入生产：

- 长上下文治理
- 状态与事件模型
- 工具调用生命周期
- 中断与恢复
- 人机协同中的控制平面

`Pi Agent Core` 的价值，不只是它提供了一个实现，而是它把这些问题提前摆到了台面上。

这也是我认为它值得认真研究的原因。
