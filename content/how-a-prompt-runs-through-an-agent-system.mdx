# 一次 Prompt 是如何在 Agent 系统里跑完的

如果想真正弄懂一个 Agent 系统，最值得盯住的，不是某个模型参数，也不是某个 Prompt 模板，而是这个问题：

> 当用户按下回车之后，这条 Prompt 到底是怎么一步一步跑完的？

很多人第一次看 Agent 代码时，容易把它想成一个“大函数”：

- 读历史记录
- 调一次大模型
- 如果有工具调用就执行
- 最后把结果显示到 UI

但真正能跑进生产环境的系统，通常不会这么写。  
它们会把产品语义、运行状态和执行循环严格拆开，因为只有这样，系统才有机会支持：

- 长上下文压缩
- 中途纠偏
- 工具执行管线
- 后续任务排队
- 可恢复的会话持久化

所以要理解“一次 Prompt 是怎么跑完的”，最核心的第一步不是记函数名，而是在脑海里先建立一道 **边界**：

这套系统绝不是把大模型请求、历史记录和 UI 揉在一个大函数里，而是有着非常严格的三层分工。

## 一、先画边界：三大核心模块分别负责什么

从代码结构上看，这三层扮演的是完全不同的角色。

### 1. `AgentSession`：产品控制面和总调度台

`AgentSession` 不负责直接跑模型，它更像是最上层的“产品控制面”。

它做的是所有带有产品语义的事情，比如：

- 解析 `/` 开头的 Slash Command
- 展开 Prompt 模板
- 加载对应的 Skill
- 处理文件引用和额外上下文
- 把底层事件同步给 UI、RPC 和持久化层

换句话说，`AgentSession` 的职责不是“回答问题”，而是 **把用户的意图整理成运行时真正能执行的任务**。

### 2. `Agent`：状态容器和运行外壳

`Agent` 更像运行时的“包工头”。

它不关心 UI 长什么样，也不关心 Slash Command 是怎么解析出来的。它接收的已经是一份相对干净的任务，然后负责把这份任务包装成一次可运行的生命周期。

它主要维护：

- 当前消息列表 `messages`
- 可用工具 `tools`
- 模型配置
- 中途纠偏用的 `steering` 队列
- 后续任务用的 `follow-up` 队列

所以 `Agent` 的本质不是一个“聊天接口”，而是一次运行过程的 **状态容器**。

### 3. `agent-loop`：真正把任务跑完的执行引擎

真正驱动整个过程的，是 `agent-loop`。

它不关心这些消息最初来自聊天框、命令面板还是模板系统，也不关心 UI 如何显示，只负责做一件事：

**在循环中不断调用模型、执行工具、回填结果，再继续调用模型，直到这次任务真的结束。**

如果把整个系统压缩成一句话：

- `AgentSession` 负责“把任务准备好”
- `Agent` 负责“把任务装进运行时状态里”
- `agent-loop` 负责“把任务一轮一轮跑完”

这三层边界一旦清楚，后面很多高级能力就都能看明白了。

## 二、一次 Prompt 的完整流转链路

接下来，我们顺着一次真实的 Prompt 生命周期往下看。

### 阶段 1：预处理与上下文重建

用户在输入框按下回车之后，最先介入的是 `AgentSession`。

这一步不是立刻把消息扔给大模型，而是先做一层“产品预处理”。

#### 1. 先解析产品语义

系统会先检查这条输入里有没有额外语义，比如：

- 是否是 Slash Command
- 是否引用了文件
- 是否命中了某个 Prompt 模板
- 是否需要挂载某个 Skill

也就是说，用户输入的并不是“原始字符串”，而是一份带有产品语义的任务描述。

#### 2. 在这里接入 Compaction

当会话已经很长时，系统不能把所有历史记录原样塞给模型，否则上下文窗口很快就会被撑爆。

这时 `AgentSession` 会在交给底层之前处理记忆问题，也就是接驳 **Compaction**。

它做的不是简单删历史，而更像：

- 把久远对话压缩成结构化摘要
- 保留最近仍然需要逐字参考的消息
- 为不同分支保留必要的总结信息

所以此时系统处理的不是“聊天记录清单”，而是经过治理后的上下文。

#### 3. 重新组装 Active Path

最终传下去的上下文，通常会只保留当前活跃路径，也就是当前正在推进的那条工作分支。

必要时，它还会合并：

- 历史压缩摘要
- 当前活动分支的近端消息
- 其他分支的总结信息

这一步很关键，因为它说明系统持有的并不是线性的聊天历史，而是 **经过重建的工作上下文**。

### 阶段 2：进入 `Agent`，初始化本轮运行状态

当 `AgentSession` 把处理好的任务交给 `Agent.prompt()` 后，运行时层才真正开始接手。

在这里，`Agent` 会做几件事情：

- 保存当前任务对应的消息状态
- 绑定可用工具和模型配置
- 初始化本轮生命周期所需的内部字段
- 重置 `steering` 和 `follow-up` 队列

到这里为止，系统做完的是“准备开工”，还没有真正进入执行循环。

## 三、核心引擎：为什么它一定是双层循环

真正复杂的部分，是 `agent-loop`。

很多人以为 Agent loop 就是一个 `while (true)`，其实成熟系统通常会把它拆成 **双层循环**：

- 内层循环：死磕当前任务，直到当前回合稳定收束
- 外层循环：检查是否还有排队的后续任务，再决定要不要继续

这个设计看起来只是多包了一层，但它决定了系统能不能自然支持 Tool Call、Steering 和 Follow-up。

### 内层循环：跑模型、调工具、响应纠偏

内层循环负责的是“把当前这件事做完”。

它大致会不断重复以下动作：

1. 把内部统一消息格式转换成模型专用协议
2. 发给底层大模型
3. 读取模型输出
4. 如果模型要求调用工具，就执行工具
5. 把工具结果回填给模型
6. 继续下一轮，直到模型输出稳定的最终文本

这就是 Tool Calling 真正落地的地方。

#### Tool Call 是怎么插进来的

当模型在某一轮输出里发起工具调用时，系统并不会结束本轮任务，而是会进入一条工具执行支路：

- 先触发 `beforeToolCall`
- 执行对应工具
- 再触发 `afterToolCall`
- 把工具结果转成消息重新喂给模型

这样一来，工具就不是外挂脚本，而是循环中的正式一环。

这套机制的意义在于，工具执行前后都可以插入业务逻辑，比如：

- 参数校验
- 权限判断
- 安全拦截
- 返回值标准化
- 审计与观测

#### Steering 是怎么中途接驳的

内层循环还有一个很关键的能力，就是支持 **Steering**。

比如模型正在输出，或者正在调用工具，这时你发现方向错了，你并不需要粗暴中断整个任务，而是可以补一句：

> 不对，用 Python 写。  
> 别继续这个方案，换一种实现。

这类消息不会简单追加到历史最末尾等待“下一次再说”，而是会先进入 `steering` 队列。

内层循环在运行时会持续检查这个队列，一旦发现有新的纠偏信息，就把它优先插入接下来的回合里，让模型尽快转向。

所以 `steering` 的本质不是“新开一轮对话”，而是 **对当前运行中的任务进行控制平面干预**。

### 外层循环：处理 Follow-up 队列

内层循环只负责当前任务收束。  
但真实使用里，用户经常会在任务快结束时顺手再补一句：

> 做完这个顺便加一组单测。  
> 结果出来以后，再帮我写个总结。

这类内容不一定适合强行插入当前推理过程，于是系统会把它们放进 **Follow-up** 队列。

外层循环的职责就是：

1. 等当前内层循环彻底结束
2. 检查 `follow-up` 队列里还有没有排队任务
3. 如果有，就再启动一轮新的内层循环
4. 如果没有，这次 Prompt 生命周期才真正结束

这样设计之后，系统就不是“一问一答”的聊天盒子，而更像一个 **可持续工作的任务线程**。

## 四、为什么 Steering 和 Follow-up 必须分开

很多系统会把“用户中途发来的新消息”粗暴地视为一类东西，但更成熟的做法，是把它们拆成两种完全不同的语义。

### `Steering`

它处理的是：

- 对当前任务立刻生效的纠偏
- 对当前回合节奏有影响的临时插话
- 需要尽快改变模型下一步行为的控制信号

### `Follow-up`

它处理的是：

- 当前任务结束后再推进的新工作
- 不需要打断当前节奏的追加任务
- 可以排队等待执行的后置动作

这两个队列的区别，看起来像实现细节，实际上是在定义 Agent 的控制平面。

如果不分开，系统很容易陷入两个问题：

- 要么用户一插话就把当前推理打乱
- 要么所有追加消息都得等整轮结束，交互显得非常迟钝

把它们拆开，系统才既能保持连续执行，又能保留人类随时纠偏的能力。

## 五、事件是怎么流出去的

到这里，模型、工具和循环都已经串起来了，但还缺最后一个问题：

**这些过程中产生的状态变化，谁来接住？**

答案还是 `AgentSession`。

虽然它不负责跑模型，但它一直像监听器一样挂在外面，持续接收底层抛出来的事件。

这些事件通常会被同步到多个地方：

- 更新内存中的会话状态
- 刷新交互界面的 UI
- 通过 RPC 广播给其他消费方
- 落盘写入 Session 持久化文件

这意味着，`AgentSession` 不是一次性地“把任务送下去就不管了”，而是持续充当整个运行时和产品层之间的数据集散中心。

## 六、Session 持久化为什么不是普通聊天记录

这里还有一个特别值得注意的点：

Session 持久化保存的，通常不是线性聊天记录，而是一棵 **可分叉的任务树**。

这件事非常重要，因为它意味着系统天然支持：

- 回到历史某个节点继续推进
- 从中间状态 `fork` 出一个新分支
- 比较不同分支上的尝试路径
- 把一次工作过程沉淀为可复用资产

从用户视角看，这只是“我可以回到之前那个点重新试一次”；  
但从架构视角看，这说明系统底层保存的是 **任务演化结构**，不是普通消息流水账。

这也解释了为什么前面 `AgentSession` 在做上下文重建时，能提取 Active Path、合并摘要、附带分支总结。因为系统从一开始就不是按“单线对话”建模的。

## 七、把整条链路连成一句话

如果把前面的过程压缩成一条完整链路，大致就是这样：

1. 用户发送 Prompt
2. `AgentSession` 解析产品语义，并处理 Compaction 和上下文重建
3. 清理后的任务交给 `Agent`
4. `Agent` 初始化本轮生命周期状态，并准备 `steering` / `follow-up` 队列
5. `agent-loop` 启动双层循环
6. 内层循环负责调用模型、执行 Tool Call、处理 Steering
7. 外层循环负责在当前任务结束后继续消费 Follow-up
8. 运行过程中产生的所有事件再被 `AgentSession` 捕获，并同步给 UI、RPC 和持久化层

一句话总结就是：

**你发出的 Prompt，先经过 `AgentSession` 做产品加工和记忆治理，再交给 `Agent` 包装成运行态，最后由 `agent-loop` 在双层循环里真正跑完；而这一过程中发生的所有动态，又会被 `AgentSession` 接回去，沉淀成可观察、可恢复、可分叉的 Session。**

## 八、为什么这个边界设计特别重要

我觉得这套设计最有价值的地方，不是“模块拆得漂亮”，而是它直接决定了系统能不能承载真实世界里的复杂性。

因为一旦三层边界清楚，很多高级能力就都有了自然落点：

- Compaction 不再是临时补丁，而是上下文重建的一部分
- Tool Call 不再是外挂逻辑，而是循环中的正式阶段
- Steering 不再只是“追加一条消息”，而是当前任务的中途控制
- Follow-up 不再只是“下一轮再说”，而是明确的排队机制
- Session 持久化 不再只是存档，而是整个任务树的恢复基础

所以真正让 Agent 系统变强的，并不只是模型本身，而是这些运行时能力有没有在架构上被认真安放。

## 结语

如果说很多文章讨论的是“Agent 能做什么”，那我更关心的是：

**Agent 是怎么把一件事稳定做完的。**

而从这个角度看，一次 Prompt 的完整生命周期，其实就是观察整个 Agent Runtime 设计成熟度的最好切口。

你会发现，真正关键的从来不是“有没有调到工具”，而是：

- 产品语义在哪里被解析
- 上下文在哪里被压缩和重建
- 运行状态由谁持有
- 循环如何终止与续跑
- 用户的中途纠偏如何接入
- 最终状态如何被观察、广播和持久化

把这些问题回答清楚，才算真正看懂了一套 Agent 系统。
